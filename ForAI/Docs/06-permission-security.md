# 权限与安全规则文档

| 浏览公开版块 | 是 | 是 | 是 | 是 |
| 浏览已发布文章 | 是 | 是 | 是 | 是 |
| 浏览评论和回复 | 否 | 是 | 是 | 是 |
| 注册登录 | 是 | 是 | 是 | 是 |
| 投稿文章 | 否 | 是 | 是 | 是 |

| 评论文章 | 否 | 是 | 是 | 是 |

| 查看自己的通知 | 否 | 是 | 是 | 是 |

## 2. 文章状态流转

状态：

- `draft`
- `pending_review`
- `published`
- `rejected`
- `archived`

MVP 默认流程：

```text
pending_review -> published
pending_review -> rejected
published -> archived
rejected -> pending_review
```

规则：

- 普通用户创建文章后默认进入 `pending_review`。
- `reviewer` 和 `admin` 可将 `pending_review` 审核为 `published` 或 `rejected`。
- 拒绝必须填写 `review_note`。
- 作者可在个人中心查看自己的 `pending_review`、`rejected` 文章。
- 作者可编辑自己的 `draft`、`pending_review`、`rejected` 文章。
- `rejected` 文章编辑保存后必须重新进入 `pending_review`，等待再次审核。
- 游客只能查看 `published` 文章。
- 已发布文章是否允许作者编辑为 TODO，默认不允许直接编辑。
- 如果未来允许编辑已发布文章，必须重新进入审核流程。

## 3. 评论与回复权限

- 游客可查看公开文章下可见评论和回复。
- 登录用户可在 `published` 文章下评论。
- 登录用户可回复可见评论。
- 普通用户可软删除自己的评论和回复。
- 普通用户不能删除或隐藏他人评论和回复。
- 审核员可隐藏或软删除任意评论和回复。
- 管理员可隐藏或软删除任意评论和回复。
- 评论和回复默认不需要审核。TODO：是否增加敏感词或审核队列后续确认。

回复通知规则：

- 评论他人文章时，后端必须创建 `article_comment` 通知。
- 回复他人评论时，后端必须创建 `comment_reply` 通知。
- 评论自己的文章时，不创建通知。
- 回复自己的评论时，不创建通知。
- 被回复评论不可见、已删除或所属文章未发布时，不允许回复。
- 通知创建必须由后端完成，前端不能直接伪造通知。

## 4. 个人中心权限

个人中心只对已登录用户开放。

普通用户可访问：

- 我的投稿。
- 我的评论和回复。
- 我的通知。
- 基础账户信息。

约束：

- 用户只能查看自己的投稿、评论和通知。
- 管理员也不通过普通个人中心查看他人私有通知；如未来需要审计能力，必须单独设计后台接口。
- 通知只允许接收人标记已读。

## 5. JWT 使用规则

- JWT 由 Go 后端签发。
- JWT 只存放在 httpOnly Cookie。
- JWT payload 必须包含：
  - `sub`：用户 ID。
  - `role`：用户角色。
  - `exp`：过期时间。
  - `iat`：签发时间。
- JWT 不能包含密码哈希、邮箱验证码、密钥等敏感信息。
- 后端必须在每次需要登录的 API 中验证 JWT。
- 后端必须从数据库确认用户仍然存在且状态为 `active`。
- 禁止仅信任前端传入的角色或用户 ID。
- TODO：是否实现 refresh token 和 token 轮换机制。

## 6. Cookie 安全规则

登录 Cookie 必须设置：

- `HttpOnly=true`
- `Path=/`
- `SameSite=Lax` 或按跨域部署需求配置为 `None`
- 生产环境 `Secure=true`
- `Max-Age` 或 `Expires` 与 JWT 过期时间一致

约束：

- 前端不读取 Cookie 内容。
- 登出接口必须清除 Cookie。
- 如果前端和后端跨域部署，必须正确配置 CORS credentials 和 Cookie Domain。
- 生产环境不得使用弱 `JWT_SECRET`。

## 7. 密码加密规则

- 密码必须使用 bcrypt 哈希后入库。
- 禁止保存明文密码。
- 禁止日志打印密码。
- 注册和修改密码时必须校验最小长度。TODO：具体密码策略后续确认，建议最小 8 位。
- 登录失败时错误提示不得区分“邮箱不存在”和“密码错误”。

## 8. 上传文件限制

第一版上传仅支持图片：

- 允许 MIME：`image/jpeg`、`image/png`、`image/webp`、`image/gif`。
- 默认最大文件大小：5MB。
- 文件名必须由后端生成，不使用用户原始文件名作为存储文件名。
- 存储路径不得包含用户可控的 `../` 等路径片段。
- 后端必须同时校验扩展名和 MIME。
- 上传目录不得允许执行脚本。
- 公开访问 URL 不应暴露服务器绝对路径。

## 9. 防越权规则

所有权限必须在后端校验：

- 修改文章时校验作者或管理员权限。
- 查看未发布文章时校验作者、审核员或管理员权限。
- 审核文章时校验 reviewer 或 admin 角色。
- 删除评论和回复时校验作者或管理角色。
- 查看通知、标记通知已读时校验通知接收人。
- 管理版块和用户角色时只允许 admin。

禁止：

- 依赖前端隐藏按钮作为唯一权限控制。
- 允许请求体传入 `authorId` 后直接信任。
- 允许普通用户通过修改 `role` 字段提升权限。
- 允许前端直接创建通知。
- 在 SQL 查询中遗漏 `deleted_at is null` 和可见性条件。

## 10. 安全响应规则

- 认证失败返回 `401 UNAUTHORIZED`。
- 权限不足返回 `403 FORBIDDEN`。
- 资源不存在或对用户不可见时可返回 `404 NOT_FOUND`，避免泄露资源存在性。
- 服务端日志可记录内部错误，但响应不得泄露堆栈、SQL、服务器路径或密钥。
