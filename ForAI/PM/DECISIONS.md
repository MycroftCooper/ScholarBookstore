# 产品决策记录

## 2026-07-21 — 将前端 UI 首轮验收前排到第 3 项

- 状态：已接受
- 决策：在产品表面收口和邀请码完成后，将全站前端 UI 首轮验收作为实施顺序第 3 项；全部功能完成后仍执行最终回归。
- 背景：项目已有大量前端页面，若等所有后端和生产工作结束后才首次统一验收，会过晚发现全局布局、导航、文案、权限入口和响应式问题。
- 理由：提前建立 UI 基线有利于集中处理前端问题；保留最终回归可覆盖首轮验收后新增或变化的功能，避免把阶段验收误认为发布验收。
- 影响：UI 工作拆成“第 3 项首轮验收”和“发布前最终回归”两次；首轮发现的未完成功能只记录待复测，不判定通过。
- 重新评估条件：无。
- 替代：只在全部功能完成后执行一次 UI 验收；已拒绝。

## 2026-07-21 — 删除独立 TODO，统一由 PM 文档维护

- 状态：已接受
- 决策：删除 `ForAI/Docs/TODO.md`。产品范围只写入 `SCOPE.md`，当前发布门槛和验收项只写入 `RELEASE.md`，交付进度和当前工作只写入 `STATUS.md`，重大取舍写入 `DECISIONS.md`。
- 背景：旧 TODO 与 PM 文档重复记录密码找回、资料、治理、PWA、存储和发布事项，已经形成第二套事实源。
- 理由：单一事实源能避免状态漂移、重复维护和已移除需求被重新实现。
- 影响：图片生命周期清理迁入当前版本 P1；外部错误追踪迁入延期范围；纯工程重构不作为长期产品待办，只有进入实际工作时才记录到 `STATUS.md`。
- 重新评估条件：无；如未来需要工程债务系统，应建立与产品范围职责不同且明确的机制。
- 替代：继续保留同步维护的 TODO；已拒绝。

## 2026-07-21 — 取消注册邮箱验证并增加最终 UI 回归

- 状态：已接受
- 决策：取消注册后的独立邮箱验证，以管理员生成的邀请码作为内测注册准入；全部范围内功能完成后，必须执行一次覆盖全站、全部角色、桌面端与移动端的前端 UI 验收回归，作为发布 P0 门槛。
- 背景：邀请制已能控制少量用户注册来源，独立邮箱验证会增加账户状态、邮件重发和权限限制复杂度；同时项目页面较多，需要在功能全部完成后统一发现视觉、交互、权限和状态展示问题。
- 理由：减少没有必要的身份流程，把质量投入转向用户实际可见的完整体验。
- 影响：删除邮箱验证 P0，密码找回仍保留邮件依赖并维持 P1；新增最终 UI 验收记录和问题闭环要求。
- 重新评估条件：改为公开注册或出现明确的邮箱真实性风险时重新评估验证流程。
- 替代：注册后强制验证邮箱；已拒绝。

- 决策：邀请码由管理员生成，单次使用，可设置有效期、手动作废并记录完整状态与使用信息；新用户注册后必须验证邮箱才能使用站内登录后能力；技术标签允许自由输入最多 10 个，不与文章 Tag 绑定；展示设置默认值为学校/地区不公开、组织/公司不公开、技术标签公开、关注与粉丝列表不公开；Dashboard 定为当前内测版本 P1。

## 2026-07-21 — 内测版本实现四项展示设置

- 状态：已接受

- 替代：实现当前四个静态开关；已拒绝。

## 2026-07-21 — 内测版本实现最小完整技术标签

- 状态：已接受
- 决策：技术标签进入当前小范围内测版本 P1。用户可以添加和删除最多 10 个标签，标签经过规范化与去重后持久化，并在个人资料页和公开作者主页一致展示。
- 背景：当前技术标签只是前端静态占位，用户无法编辑或保存。
- 理由：产品负责人确认技术标签具有作者资料价值，但应控制范围，避免演变成推荐和能力认证系统。
- 影响：需要补数据存储、资料 API、编辑交互、公开展示和测试；明确不建设基于标签的推荐、排名、认证、擅长度或工作年限。
- 重新评估条件：真实用户需求证明需要标签发现或推荐能力。
- 替代：上线后再做并在首发隐藏；已拒绝。

## 2026-07-21 — 首发采用邀请制小范围内测

- 状态：已接受
- 决策：首批面向约 10–30 名邀请用户，不以公开注册为首发目标。
- 背景：公开注册会立即引入邮件验证、密码找回、垃圾用户和更复杂风控需求。
- 理由：邀请制符合少量用户定位，并允许用简单人工运营换取更低发布复杂度。
- 影响：邀请流程成为发布门槛；最终决策不做注册邮箱验证，密码找回仍为 P1，复杂自动风控继续延期。
- 重新评估条件：内测稳定且需要扩大获客范围。
- 替代：直接公开注册；当前拒绝。
